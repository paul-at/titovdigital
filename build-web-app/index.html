The Pipeline

Contracting web application development is a perilous task. There is always uncertainty whether the result will function and appear as desired; whether the deadlines will be met; and the biggest risk, whether the entire application would have to be redone because of a scoping mistake.

There is a saying in project management that for each project cost, time and scope are mutually exclusive. However, with the right execution the characteristics of the result are predetermined and can be relied upon. Titov Digital uses a four-step web application delivery pipeline that eliminates the uncertainty risks.

1. Value and Pain

Step one defines how the application will solve users' pain and create value by automating existing process.

While this document focuses on web applications in general, you may wish to read ones about specific types of applications: SaaS web applications, business automation and workflow applications, financial API integrations and marketing API integrations.

Or have a look at our homepage to read more about what we do.

A well designed web application makes a process five to ten times faster for users, enabling new customer and employee interactions. For example:

Simple access to information. A web application helps customers and employees to interact with the data by eliminating manual form processing.

Data analysis. A web application collects vast amounts of user data automatically and processes them continiously, uncovering insights that are hard or impractical to discover using manual reporting.

Ease of interaction and Scale. Automating a manual process allow to reach larger audience, and guiding customer interactions inside an app makes them more likely to engage.

Data integrity. A web application enforces validity of database records continiously, eliminating mistakes and data losses in spreadsheets.

2. Project Scope

At this step, a high-level specification is produced, describing the functions new application performs at the high level.

The goal of this step is to have an understanding of how much resources and time each possible feature requires and select ones that will be included in the first version, a minimum viable product.

Together with non-functional requirements and problem definition prepared in the previous step these requirements are recorded in a Statement of Work.

Read more about approaching web application MVP in Our Dangerous Obsession With The MVP by Bill Aulet.

3. Development and Acceptance Testing

At Titov Digital, development and user acceptance testing are conducted simultaneously. Each weak or fortnight, depending on the project, development cycle completes and the results, in a form of finished features or milestones are presented at a demo.

The feedback from the demo is implemented the following week. This process closes loop between requirements and application as tightly as possible, permitting minor changes and improvements that couldn't have been predicted in the planning stage.

4. Commissioning

The application is deployed to cloud hosting infrastrcture and the code along with necessary documentation is handed over.
